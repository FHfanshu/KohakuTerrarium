# skills、info 和 skill_mode 进阶说明

这一页主要讲一件事：在 KT 里，**skill 不是工具本体，而是给模型看的“使用说明”**。

很多人第一次看到 `skills`、`info`、`skill_mode` 这几个词，会很容易把它们和“工具开关”“插件系统”“MCP 连接”“安装包”混在一起。其实不是一回事。这篇就用大白话掰开说。

## 先一句话讲明白

你可以先这样记：

- **工具 / 子智能体**：真正能干活的东西
- **skill**：告诉模型“这个东西怎么用”的文档
- **info**：按名字去把这份文档读出来
- **skill_mode**：决定这份文档是“一开始就塞进 prompt”，还是“需要时再读”

如果你先把这四个东西分开，后面基本就不容易绕晕。

## KT 里的 skill，到底是什么

KT 里的 skill，和很多别的框架说的 skill，不是一个路子。

在别的框架里，skill 常常指的是“一个可调用能力单元”，比如某个函数、某个工具、某种 action。

但在 **KohakuTerrarium** 里，**skill 更像是一份说明书**。

也就是说：

- 它不是能力本身
- 它不是 Python 函数
- 它不是 MCP 服务
- 它也不是安装包
- 它是“写给模型看的文档”

你可以把它理解成：

> 工具有没有，是一回事；模型知不知道该怎么正确用这个工具，是另一回事。skill 管的是后者。

## 和其他框架的区别

下面这个表可以快速对比一下：

| | 其他框架 | KohakuTerrarium |
|---|---|---|
| 它到底是什么 | 可调用能力单元 | 给模型看的说明文档 |
| 里面装的内容 | Python 函数 + JSON Schema | `builtin_skills/` 下面的 `.md` 文档 |
| 怎么用 | 规划后直接调用 | 通过 `info` 按需读取，或者由框架直接注入 prompt |
| 通常放哪 | Python 模块 | `src/kohakuterrarium/builtin_skills/tools/*.md` |

所以你看到 KT 里的 `skill`，不要本能地理解成“这个功能本身”。

更准确的理解应该是：

**skill = 模型的工具说明书 / 操作手册**。

## builtin_skills 是什么

`builtin_skills` 就是框架自带的一批说明文档。

常见位置是这些目录下面的 `.md` 文件：

- `src/kohakuterrarium/builtin_skills/tools/`
- `src/kohakuterrarium/builtin_skills/subagents/`

注意，这里存的是 **Markdown 文档**，不是某种要安装的插件包。

所以你可以把 `builtin_skills` 理解成：

> KT 自带的“模型操作手册库”

它的作用是让模型在需要的时候，知道某个工具怎么用、什么时候用、参数怎么传、有哪些坑要避开。

## info 是什么

`info` 是一个用来**按名称读取完整说明文档**的工具。

表面上看，它像“查帮助”。
但如果你只把它理解成“哦，就是帮助命令”，那就有点低估它了。

在 `dynamic` 模式下，`info` 的作用很关键，因为模型一开始拿到的往往只是简短描述，不是完整手册。这时候如果要更稳、更准地调用工具，就得先通过 `info` 把完整文档补读出来。

所以更接近实际的理解是：

> 在 dynamic 模式下，`info` 是模型“先翻说明书再动手”的入口。

也就是说，它不是可有可无的小配角，而是工作流里很重要的一环。

## skill_mode 是什么

`skill_mode` 控制的是：

**这些 skill 文档到底怎么进入 prompt。**

重点来了，它控制的只是“文档怎么给模型看”，**不是**：

- 不是工具开关
- 不是 MCP 连接开关
- 不是 package 发现机制
- 不是“有没有安装某个功能”的判断方式

也就是说，`skill_mode` 不决定一个工具存不存在，只决定模型什么时候看到那份说明书。

配置长这样：

```yaml
skill_mode: dynamic  # 默认：先给短描述，需要时再用 info 补读完整文档
skill_mode: static   # 直接把完整文档放进系统提示词
```

## dynamic 和 static，到底差在哪

### 1）dynamic

`dynamic` 是默认模式。

这种模式下，模型一开始不会拿到所有 skill 的完整内容，通常只会先看到比较短的描述。等它发现“这个工具我得仔细看看怎么用”，再去调 `info` 读完整文档。

好处很直接：

- prompt 更短
- 更省上下文
- 工具多的时候没那么占 token
- 需要哪个再读哪个，比较灵活

代价也很明显：

- 可能要多一次 `info` 调用
- 如果模型不主动读文档，调用质量可能会受影响

一句大白话总结：

> dynamic 就像桌上只先摆目录，真要用的时候再翻对应章节。

### 2）static

`static` 则相反。

这种模式下，完整 skill 文档会在一开始就直接进系统提示词。也就是说，模型开局就把相关说明都看到了。

好处是：

- 上来就有完整上下文
- 不用额外再调用 `info` 才知道细节
- 更适合要求固定、稳定、可审计的提示词场景

代价是：

- prompt 会更大
- 工具一多就更吃上下文
- 有些文档模型可能根本用不上，但也已经塞进去了

一句话说：

> static 就像开工前先把整本操作手册发到你手里。

## 最容易搞混的几个误区

## 误区 1：skill_mode 是功能开关

不是。

`skill_mode` 不决定工具能不能用。

一个工具能不能用，通常看这些事：

- 这个 creature 有没有把它暴露出来
- 对应的 MCP 有没有连上
- 相关 package 有没有安装好
- 当前配置有没有真的启用它

而 `skill_mode` 管的是：

- 这个工具的说明文档，是开局直接给模型
- 还是让模型需要时自己通过 `info` 去读

所以别把它理解成“开了 static 工具就更全了”或者“开了 dynamic 某些能力就没了”。

**能力集合不一定变，变的是文档注入方式。**

## 误区 2：info 只是查帮助

也不完全对。

从功能表面上说，它当然是在读帮助文档。

但在 `dynamic` 模式下，它会直接影响模型到底能不能把工具用对。因为模型如果只拿到很短的简介，很多细节并不知道，这时先调 `info` 再行动，往往才是更稳的做法。

所以你可以把它看成：

> “需要精确操作时，先补读说明书”的标准入口。

## 误区 3：skills 等于 package

不是一回事。

- **package** 讲的是内容怎么分发、怎么安装
- **skill** 讲的是能力说明怎么提供给模型

package 更像“货是怎么送来的”，skill 更像“货到了以后附带的说明书”。

这两个概念可以有关联，但不能直接画等号。

## 一个实际例子

看下面这个配置：

```yaml
name: docs-aware-agent
version: "1.0"
base_config: "@kt-defaults/creatures/swe"

skill_mode: dynamic

tools:
  - info
  - read
  - grep
```

如果提示词里再写一句：

```md
不确定工具参数时先调用 info。
```

那这个 agent 的工作习惯就会比较清楚：

1. 先知道自己能用哪些工具
2. 如果某个工具细节不确定，就先调 `info`
3. 看完说明再去 `read`、`grep` 之类的实际操作

这里要特别注意：

如果你把

```yaml
skill_mode: dynamic
```

改成

```yaml
skill_mode: static
```

**变化的重点是“文档怎么给模型”变了，不一定是“工具列表”变了。**

也就是说，`info`、`read`、`grep` 这些工具本身是不是可用，主要不是由 `skill_mode` 决定的。

## 什么时候更适合用 static

如果是下面这些情况，`static` 往往更合适：

- 工具数量不多
- 你希望模型一开始就看到完整说明
- 你不想让模型每次再去补读文档
- 你需要比较固定、可审计的 prompt 内容

尤其是在一些对行为稳定性、提示词审计要求比较高的场景里，`static` 会更省心一点。

## 什么时候更适合用 dynamic

如果是这些情况，通常更适合 `dynamic`：

- 工具很多
- 上下文预算比较紧
- 不想把所有文档一股脑塞进 prompt
- 希望模型按需加载说明
- 愿意接受“必要时先调一次 info”这个流程

简单说：

- 文档少、想一次看全：`static`
- 文档多、想省上下文：`dynamic`

## 这三个层次最好分开看

理解 KT 这套东西时，一个很有用的办法，就是把下面三层拆开：

### MCP

MCP 解决的是：**能力从哪里来**。

也就是某些功能是不是通过外部服务接进来的。

### package

package 解决的是：**内容怎么安装、怎么分发**。

也就是某些组件是不是通过安装包提供的。

### skills

skills 解决的是：**怎么把这些能力的说明交给模型**。

也就是模型知不知道这些东西该怎么正确使用。

所以最核心的一句话是：

> “能力来源”和“文档暴露方式”是两码事。

你把这件事想清楚了，`skills`、`info`、`skill_mode` 基本就不会再混了。

## 最后再用一句话收尾

如果你还想再压缩成一句最容易记住的话，那就是：

> 在 KT 里，skill 更像说明书，不像功能按钮；`info` 是翻说明书；`skill_mode` 是决定说明书什么时候发给模型。
