# skills、info 和 skill_mode 进阶说明

先看结论：

- **工具 / 子智能体**：真正执行任务的能力
- **skill**：写给模型看的使用说明，告诉它何时、按什么步骤使用这些能力
- **info**：按名称读取某份完整说明文档
- **skill_mode**：决定说明文档是启动时直接注入，还是运行中按需读取

最容易混淆的一点是：**skill 不是工具本体，skill_mode 也不是功能开关。**

在 KT 里，可以直接这样记：

> 工具负责干活，skill 负责教模型怎么干；`info` 负责翻说明书，`skill_mode` 负责决定说明书何时发给模型。

---

## 先把几个概念分开

### skill 不是什么

skill 不是：

- 工具开关
- MCP 连接
- 安装包
- Python 函数
- 模型权重或微调结果

### skill 是什么

skill 本质上是**面向模型的任务说明**。它通常会说明：

- 这个 skill 适合什么场景
- 应该调用哪些工具或子智能体
- 推荐的执行步骤
- 参数注意事项、限制和常见坑

所以，skill 解决的是：**模型知不知道该怎么正确使用能力。**

### 和相邻概念的区别

| 概念 | 作用 |
|---|---|
| 工具 / MCP | 提供实际能力接口 |
| 子智能体 | 作为独立角色执行某类任务 |
| skill | 提供某类任务的使用说明或 SOP |
| package | 负责安装、分发内容 |

一句话区分：

- **工具 / MCP / 子智能体**：能力本身
- **skill**：能力的说明文档
- **package**：内容的分发方式

---

## 业界里说的 skills，通常指什么

在 Claude Code、OpenCode、OpenClaw 等框架里，skills 一般都可以理解为：**给 agent 复用的任务模块或任务说明包**。

它通常不是一句临时 prompt，而是一份结构化文档，甚至可能带有模板、脚本和依赖配置。常见内容包括：

- 名称和用途
- 适用场景
- 推荐流程
- 配套资料或文件

所以它和普通 prompt、rules、工具并不是一回事。

### 和普通 prompt 的区别

- **prompt**：一次性指令
- **skill**：可复用、可检索、带结构的任务说明

### 和 rules / AGENTS.md / CLAUDE.md 的区别

- **rules** 更偏通用约束和长期行为规则
- **skill** 更偏某类任务的专门流程

如果一段内容已经从“行为约束”变成“执行某类任务的操作手册”，更适合做成 skill。

### 和 tool / MCP 的区别

- **tool / MCP**：提供能力
- **skill**：说明能力怎么用

可以把它理解成：

- tool / MCP = 手和脚
- skill = 操作手册

### 和 agent 的区别

- **agent**：完整助手角色，可能有独立 prompt、权限、模型
- **skill**：agent 可按需调用的技能说明

---

## KT 里是怎么落地的

KT 沿用了业界的思路，但做法更轻。

在 KT 里，skill 目前主要是**给模型看的 Markdown 文档**，重点不是封装独立脚本，而是补足复杂工具的使用说明。

也就是说，KT 的 skill 更接近：

> 工具说明书 / 操作手册

而不是“可执行插件包”。

### 为什么需要它

很多工具的函数签名和简短描述不足以覆盖这些信息：

- 参数细节
- 使用时机
- 推荐步骤
- 常见误用
- 失败场景的处理方式

KT 的 skill 就是把这些内容整理成文档，供模型在需要时读取。

### KT 中的 skill 长什么样

常见位置：

- `src/kohakuterrarium/builtin_skills/tools/`
- `src/kohakuterrarium/builtin_skills/subagents/`

这里存放的是 `.md` 文档，不是插件安装包。

所以 `builtin_skills` 更准确的理解是：

> KT 自带的模型说明书库。

### 和其他框架相比，KT 的差异

| | 其他框架 | KohakuTerrarium |
|---|---|---|
| 重点 | 任务技能包，可能带脚本和依赖 | 模型说明文档 |
| 内容形式 | `SKILL.md` + 代码 + 配置 | `builtin_skills/` 下的 `.md` |
| 作用 | 帮 agent 完成某类任务 | 帮模型正确使用工具或子智能体 |

因此，看到 KT 里的 `skill` 时，优先把它理解为：**说明文档，不是功能本体。**

---

## `info`：按需读取完整说明

`info` 的作用很直接：**按名称读取某份完整 skill 文档。**

它表面上像“查看帮助”，但在 KT 里，它通常还承担一个更实际的职责：

> 当模型手头只有简短描述、不足以安全调用工具时，先补读完整说明。

这也是为什么 `info` 在 `dynamic` 模式下尤其重要。

### 什么时候应该先用 `info`

例如这些情况：

- 不确定参数含义
- 工具有前置步骤或限制
- 该工具比较复杂，误用成本高
- 只看短描述不足以判断正确用法

这时先读文档，再执行工具，通常更稳。

---

## `skill_mode`：决定说明文档如何进入 prompt

`skill_mode` 控制的是：**skill 文档以什么方式提供给模型。**

它不控制：

- 工具是否存在
- MCP 是否连接成功
- package 是否安装完成
- 某个能力是否被启用

所以不要把 `skill_mode` 理解成“功能开关”。它只影响**说明文档的注入方式**。

配置示例：

```yaml
skill_mode: dynamic  # 默认：先给短描述，需要时再用 info 读取完整文档
skill_mode: static   # 启动时直接注入完整文档
```

### `dynamic`

默认模式。

模型一开始通常只拿到简短描述；如果需要更多细节，再调用 `info` 读取完整文档。

优点：

- prompt 更短
- 更省上下文
- 工具很多时更灵活

代价：

- 可能多一次 `info` 调用
- 如果模型没先读文档，调用质量可能下降

一句话：**先看目录，需要时再翻手册。**

### `static`

完整 skill 文档在启动时就进入系统提示词。

优点：

- 模型开局就有完整说明
- 不需要额外调用 `info` 才知道细节
- 更适合固定、可审计的提示词场景

代价：

- prompt 更大
- 工具多时更占上下文
- 可能提前注入很多实际用不到的文档

一句话：**开工前先把整本手册发下去。**

---

## 一个最常见的误区

### 误区：切换 `skill_mode`，就是在切换功能是否可用

不是。

`skill_mode` 改变的是：

- 文档开局直接给模型
- 还是运行中由模型通过 `info` 按需读取

它不直接决定：

- `read`、`grep`、`info` 这些工具是否可用
- 某个 MCP 是否接入成功
- 某个 package 是否已经安装

核心区别只有一句话：

> **能力来源** 和 **文档暴露方式** 不是一回事。

## 一个实际配置例子

```yaml
name: docs-aware-agent
version: "1.0"
base_config: "@kt-biome/creatures/swe"

skill_mode: dynamic

tools:
  - info
  - read
  - grep
```

如果再补一条规则：

```md
不确定工具参数时先调用 info。
```

这个 agent 的执行流程就会变成：

1. 先知道有哪些工具可用
2. 遇到不确定的工具，先用 `info` 读完整说明
3. 再调用 `read`、`grep` 等工具执行实际任务

如果把：

```yaml
skill_mode: dynamic
```

改成：

```yaml
skill_mode: static
```

变化点主要是：**说明文档的提供方式变了。** 工具列表本身不一定变化。

---

## 该怎么选：`static` 还是 `dynamic`

### 更适合 `static` 的场景

- 工具数量不多
- 希望模型一开始就看到完整说明
- 不想依赖运行中再读文档
- 需要更固定、可审计的 prompt

### 更适合 `dynamic` 的场景

- 工具很多
- 上下文预算紧
- 不想把所有说明文档都塞进 prompt
- 接受模型按需先调用一次 `info`

可以直接记成：

- **文档少、想一次看全**：`static`
- **文档多、想省上下文**：`dynamic`

---

## 最后再压缩成一句话

在 KT 里：**skill 是说明书，`info` 是读说明书，`skill_mode` 是决定说明书何时发给模型。**
